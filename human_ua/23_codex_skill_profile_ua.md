# Профіль роботи Codex для цього проєкту

## Головне правило

Codex має писати код так, щоб його можна було:
- запустити;
- перевірити;
- зрозуміти;
- підтримувати.

## Як Codex має працювати

### 1. Думати вертикальними slices
Кожен етап повинен давати реальний приріст:
- endpoint;
- worker path;
- browser flow;
- UI interaction;
- host action;
- test.

### 2. Писати пояснювальні коментарі
Коментарі потрібні не для кожного рядка, а для:
- архітектурних рішень;
- неочевидних контрактів;
- state transitions;
- timeout/retry логіки;
- file transfer;
- browser sync points;
- host action safety limits.

Коментар має пояснювати:
- **що компонент робить**;
- **чому він зроблений саме так**;
- **де межі його відповідальності**.

### 3. Не лінуватися на завершення
Codex не повинен:
- залишати “дописати потім” у критичних місцях;
- зупинятися після scaffold без робочого execution path;
- робити вигляд, що інтеграція є, коли вона лише заглушена.

### 4. Писати ясний код
Очікування:
- зрозумілі імена;
- невеликі функції;
- чіткі dataclass/Pydantic contracts;
- явні error classes;
- один спосіб робити одну річ;
- мінімум магії.

### 5. Тримати тести корисними
Тести мають перевіряти:
- контракти;
- state transitions;
- serialization;
- retry/lease logic;
- failure handling;
- integration seams;
- критичні utility paths.

Тести не повинні:
- просто повторювати реалізацію;
- перевіряти дрібниці без цінності;
- бути шумом заради відсотка покриття.

## Що вважати успішною роботою Codex

- можна розгорнути систему за інструкцією;
- немає критичних моків у основних шляхах;
- можна пройти acceptance suite;
- код можна читати як навчальний матеріал.
