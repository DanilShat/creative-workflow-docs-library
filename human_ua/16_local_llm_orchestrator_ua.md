# Локальна LLM оркестратора

## Роль локальної моделі

Локальна модель потрібна не для генерації медіа, а для дешевої логіки orchestration.

Вона має вміти:
- нормалізувати brief у структуру;
- класифікувати задачу (`static`, `video`, `photoshop_action`, `aftereffects_action`);
- пропонувати наступний крок workflow;
- збирати structured payload для jobs;
- підсумовувати фідбек дизайнера;
- вирішувати, коли потрібен retry, repair, approval або escalation.

## Чого локальна модель не робить

- не оцінює якість відео як головний суддя;
- не заміняє Gemini prompt-building там, де потрібен сильніший зовнішній інструмент;
- не виконує довільний код у Photoshop/AE;
- не має прямого доступу до браузерів воркера.

## Правильний режим роботи

Локальна модель працює як **default orchestrator brain**.

Fallback-режим:
- якщо завдання надто складне для локальної моделі;
- якщо потрібен кращий prompt-repair;
- якщо потрібен багатший prompt expansion;

тоді сервер може створити browser job на Gemini.

## Що важливо для стабільності

Щоб локальна модель реально працювала на ноутбуці, її треба використовувати економно:
- короткий system prompt;
- JSON output schema;
- мінімум довгої історії;
- summaries замість сирих переписок;
- без heavy multimodal reasoning.

## Архітектурне правило

Локальна LLM повинна бути доступна **лише серверу**.

Воркер не ходить до неї напряму. Це важливо для:
- безпеки;
- простішої мережевої схеми;
- централизації логіки;
- контролю версії model policy.
