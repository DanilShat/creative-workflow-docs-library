# План розгортання і перевірки для власника

## Загальна ідея

У системі є два вузли:

### Ноутбук власника
Тут працюють:
- FastAPI server;
- PostgreSQL;
- LangGraph orchestration;
- local LLM runtime;
- центральне сховище метаданих.

### Ноутбук дизайнера
Тут працюють:
- worker service;
- Playwright browser execution;
- Photoshop companion;
- After Effects bridge;
- локальні тимчасові файли.

## Що підготувати заздалегідь

### На своєму ноутбуці
- Python
- PostgreSQL
- Git
- локальний runtime для LLM
- каталог для storage/artifacts
- відкритий порт для server API у локальній мережі

### На ноутбуці дизайнера
- Python
- браузер для Playwright
- Photoshop
- After Effects
- доступ до сервера по локальній мережі
- окрема папка під worker temp/cache

## Порядок першого розгортання

### Крок 1. Підняти сервер
Перевірити:
- API стартує;
- БД доступна;
- міграції застосувалися;
- local LLM відповідає;
- health endpoint працює.

### Крок 2. Підготувати worker
Перевірити:
- worker читає конфіг;
- знає URL сервера;
- може зареєструватися;
- може слати heartbeat;
- може приймати job.

### Крок 3. Підготувати браузерні профілі
На машині дизайнера:
- запустити worker у режимі setup;
- відкрити Gemini / Freepik;
- увійти вручну;
- зберегти профілі;
- перевірити, що повторний запуск профілю не просить логін.

### Крок 4. Підготувати Photoshop/AE інтеграції
- встановити Photoshop side panel;
- перевірити локальний виклик bridge;
- перевірити, що host action запускається;
- перевірити повернення статусу.

### Крок 5. Запустити перший реальний E2E
Сценарій:
1. створити задачу;
2. прикріпити refs;
3. local LLM розбирає brief;
4. воркер йде в Gemini;
5. воркер будує prompt;
6. воркер йде у Freepik;
7. воркер генерує картинку;
8. артефакт повертається на сервер;
9. UI показує результат;
10. людина робить approve або reject.

## Що обов'язково перевірити

- чи бачить сервер воркер;
- чи живий heartbeat;
- чи не дублюються jobs;
- чи зберігаються артефакти у правильну папку;
- чи не губляться файли;
- чи видно логи;
- чи після перезапуску система не “забуває” активний стан.

## Мінімальний критерій готовності

Система готова до першого реального використання, якщо:
- один дизайнер може створити задачу;
- один воркер може її виконати;
- один browser flow реально працює;
- один Photoshop action реально працює;
- після помилки видно, що саме зламалося.
