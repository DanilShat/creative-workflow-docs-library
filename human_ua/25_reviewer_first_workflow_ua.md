# Reviewer-first workflow перед розробкою

## Навіщо це потрібно
Перед тим як просити Codex писати код, краще запустити окремий чат у режимі рев'юера. Його задача — подивитися на всі спеки як на єдину систему і знайти тільки ті прогалини, які можуть зламати реальний MVP.

Це не ще один етап фантазування. Це захист від ситуації, коли Codex читає багато документів, неправильно розв'язує суперечності й будує красивий каркас, який запускається, але не працює end-to-end.

## Що має зробити reviewer-chat
Він має перевірити:

- чи зрозуміло, що запускається на твоєму ноуті, а що на ноуті дизайнерки;
- чи локальна LLM використовується тільки як оркестратор;
- чи браузерні сервіси виконуються саме на машині дизайнерки;
- чи є достатньо чіткий worker protocol;
- чи є перший реальний vertical slice без моків;
- чи не залишилось старих API-first формулювань, які конфліктують із browser-first дизайном;
- чи не залишилось небезпечних ідей типу виконання довільного коду в Photoshop/After Effects;
- чи достатньо чіткі acceptance criteria.

## Як запускати
У Codex відкрий цю папку як репозиторій/проект.

Першим повідомленням дай промпт:

- `prompts/00_reviewer_first_system_review.md`

Після відповіді рев'юера:

1. Подивись на `Critical blockers`.
2. Якщо вони є — не проси писати код.
3. Дай другий промпт:
   - `prompts/00_reviewer_gap_closure_patch_request.md`
4. Попроси Codex внести патчі в документацію або чітко сформулювати, які файли треба змінити.

## Що очікувати від рев'юера
Хороший результат виглядає так:

- короткий verdict: `Ready`, `Almost ready`, або `Not ready`;
- таблиця критичних блокерів;
- список важливих, але не блокуючих gaps;
- карта суперечностей у спеках;
- перевірка першого real MVP vertical slice;
- точний список документів, які треба оновити.

## Коли переходити до реалізації
Переходь до реалізації тільки коли reviewer-chat каже:

- немає критичних блокерів;
- перший vertical slice зрозумілий;
- server/worker protocol достатньо чіткий;
- browser-first і DCC allowlisted action рішення не конфліктують зі старими документами;
- runbook достатній для першого локального запуску.

## Що не треба робити на цьому етапі
Не треба просити рев'юера:

- писати код;
- переробляти всю архітектуру;
- додавати cloud deployment;
- розширювати scope;
- оптимізувати UI;
- робити повну автоматизацію After Effects.

Reviewer-first етап потрібен тільки для того, щоб Codex не почав будувати неправильну або мокову систему.
